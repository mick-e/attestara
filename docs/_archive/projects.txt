Technical Architecture Assessment
What's genuinely strong
The three-layer protocol design is correct and elegant. Credential Layer, ZK Proof Layer, and Commitment Layer have clean separation of concerns. Each solves a specific dimension of the governance gap without overreaching. This is good systems design.
The ZK circuit selection is defensible. Groth16 over PLONK is the right call for the current stage — the gas advantage (~210K vs ~300K+) is material at per-session economics, and the trusted setup risk is manageable with a proper multi-party ceremony. The four circuits — MandateBound, ParameterRange, CredentialFreshness, IdentityBinding — cover the core mandate enforcement use cases without unnecessary complexity.
The W3C VC 2.0 alignment is ahead of the market. Most projects in this space are still building on VC 1.1. Using the May 2025 recommendations as the foundation means Attestara won't need a painful credential migration when the ecosystem catches up. The eddsa-jcs-2022 cryptosuite choice over secp256k1 is technically correct and the reasoning in the whitepaper is sound.
The Arbitrum L2 cost model is compelling. $0.06-$0.08 per session versus $48 on Ethereum mainnet is the difference between a theoretically interesting protocol and an economically viable one. This is one of the most important commercial decisions in the entire architecture and it's been made correctly.
The regulatory alignment is unusually thorough. The explicit mapping to EU AI Act Articles 9, 12, 14, and 17, with specific JTC 21 working group alignment, is exactly what a CISO or compliance team needs to justify procurement. Most technical protocols treat regulatory alignment as an afterthought. Attestara has made it structural.

Technical weaknesses that need addressing
The CredentialFreshness oracle problem is the most serious unresolved technical issue. The whitepaper acknowledges it partially but doesn't fully resolve it. The circuit proves a credential is valid at time T, but a ZK circuit cannot query on-chain state during proof generation. The Merkle proof approach with a 90-second freshness window is a reasonable mitigation, but the SHA-256 bridging gadget adding ~6,000 constraints is expensive and the trust assumptions around the status list root anchoring are not fully specified. A sophisticated technical reviewer at Trail of Bits will flag this immediately.
Specifically: who triggers the status list root update on-chain? What happens if the anchoring service is down? The 90-second window means a revoked credential can still generate valid proofs for up to 90 seconds after revocation. For high-value financial commitments that's a meaningful attack window.
The IdentityBinding circuit fallback is underspecified. The whitepaper mentions a fallback where if Week 5 benchmarks show proof generation exceeding 5 seconds, you switch to an off-chain signature verification approach. But this fallback fundamentally changes the security model — you're replacing a ZK guarantee with a trust assumption. The conditions under which the fallback activates, and the security implications of doing so, need a full specification before any enterprise deployment.
The managed prover TEE architecture has a trust gap. The whitepaper states that mandate parameters are decrypted inside a TEE (Intel TDX or AMD SEV-SNP) and never logged outside it. TEE remote attestation is described as "available on demand." But for a financial services firm deploying this under DORA, "available on demand" is not sufficient — they need continuous attestation with audit logs proving the TEE integrity was maintained throughout every session. The enterprise deployment model needs a more robust attestation framework.
The cross-org session design gap identified in earlier design review is still present. The relay auth, WebSocket channels, and database schema are described as org-scoped, but the most commercially valuable use case — Agent A from Org 1 negotiating with Agent B from Org 2 — requires cross-org trust establishment that isn't fully specified in the session protocol. The TurnProofBundle exchange is clear, but the initial session authorisation handshake for genuinely adversarial cross-org contexts needs more detail.
Circuit versioning is unaddressed. This is a practical production issue that will matter immediately. If a circuit bug is found after mainnet deployment, what happens to existing commitment records that reference the old verifier contract? The upgrade path for the CommitmentContract when circuit versions change is not described. For a protocol making legally admissible commitment records, this is not a minor detail.

Understanding The Problem Space First
Before writing the specification, let's be precise about what the anchoring service actually needs to do and why each requirement exists.
The anchoring service has three distinct responsibilities that are often conflated:
Responsibility 1 — Bitstring Status List Monitoring. Watching the W3C Bitstring Status List credential for revocation events and detecting when the status of any registered credential changes.
Responsibility 2 — Merkle Root Computation. Computing the Poseidon Merkle root over the current bitstring state and maintaining the SHA-256 bridge for on-chain compatibility.
Responsibility 3 — On-Chain Anchoring. Submitting the computed Merkle root to the CredentialRegistry contract on Arbitrum with appropriate timing, gas management, and confirmation handling.
Each responsibility has different failure modes and different recovery behaviours. Conflating them in the specification creates ambiguity that auditors will flag and enterprise customers will question.

The Core Design Decisions To Make First
Before writing the specification, four architectural decisions need to be made explicitly. Each has real implications for security, cost, and operational complexity.

Decision 1 — Centralised vs Decentralised Anchoring
Option A — Littledata operates the anchoring service.
Single operator, simpler architecture, faster to deploy. The anchoring service is a managed component like the relay. DORA-subject institutions can self-host it.
Option B — Any participant can anchor.
The anchoring service is a permissionless component — any party can submit a valid Merkle root to the CredentialRegistry. The contract validates correctness and accepts the first valid submission in each window.
Option C — Threshold multi-party anchoring.
Multiple independent anchoring nodes must agree on the Merkle root before it's accepted on-chain. More resilient but significantly more complex.
Recommendation: Option A for v1 with self-hosting capability for enterprise customers. Option B creates a validation complexity problem — the contract needs to verify the Merkle root computation on-chain which is expensive. Option C is the right long-term architecture but premature for the current stage. Document the migration path from A to C in the specification so auditors and enterprise customers can see the decentralisation roadmap.

Decision 2 — Event-Driven vs Time-Driven Anchoring
Option A — Pure time-driven. Anchor every N seconds regardless of whether any credential status has changed. Simple, predictable, but wastes gas when nothing has changed.
Option B — Pure event-driven. Anchor only when a revocation event is detected. Minimises gas cost but creates variable latency — if no revocations happen for hours, the on-chain root becomes stale relative to the freshness window.
Option C — Hybrid. Anchor on every revocation event plus a heartbeat anchor every N seconds to keep the root fresh even when no revocations occur.
Recommendation: Option C is clearly correct. The heartbeat interval should be configurable per deployment with the default set conservatively at 60 seconds. The revocation event trigger ensures immediate response to the most security-critical events. Document the cost model for both triggers explicitly — the whitepaper already shows ~$0.04 per root update, so at 60-second heartbeat intervals the daily anchoring cost is approximately $57.60 which is acceptable for enterprise deployments.

Decision 3 — Freshness Window Enforcement
The current specification says the on-chain verifier must confirm the status_list_root was anchored within a configurable freshness window with a default of 90 seconds.
The problem is that 90 seconds creates an attack window. A credential revoked at T=0 can still generate valid proofs until T=90 seconds. For high-value financial commitments that's a meaningful exposure.
The decision is whether the freshness window is:
A — Protocol-level fixed. One freshness window for all deployments, set conservatively.
B — Deployment-level configurable. Each enterprise deployment sets their own freshness window based on their risk tolerance and operational constraints.
C — Commitment-level configurable. The freshness window is specified per commitment type — tighter for high-value financial transactions, looser for standard procurement.
Recommendation: Option C with sensible defaults. Define three tiers in the specification:

High security — 30 seconds. For financial services commitments above a defined materiality threshold.
Standard — 90 seconds. Default for most enterprise procurement use cases.
Low frequency — 300 seconds. For low-risk negotiation environments where anchoring cost is a constraint.

The tier is specified in the Authority Credential at issuance time, not negotiated between agents. This means the Principal determines the security level when issuing the credential, which is the correct governance model.

Decision 4 — Failure Mode Behaviour
What happens when the anchoring service is unavailable and the on-chain root becomes stale beyond the freshness window?
Option A — Hard fail. Sessions cannot proceed if the status_list_root is stale. All negotiation turns are rejected until fresh anchoring resumes.
Option B — Graceful degradation. Sessions can proceed with a degraded trust level, communicated to both parties, for a defined grace period beyond the freshness window.
Option C — Session suspension. Active sessions are suspended rather than failed — turns are queued and replayed when anchoring resumes, provided the session anchor hasn't expired.
Recommendation: Option A for high security tier, Option C for standard and low frequency tiers. Hard failure for financial services commitments is the correct behaviour — a DORA-subject institution should not be making commitments under uncertain revocation status. Session suspension for standard procurement makes the protocol more resilient without compromising security for lower-risk use cases.

The Anchoring Service Specification
With those decisions made, here is the full specification.

Section X.1 — Overview
The Attestara Anchoring Service is the operational component responsible for maintaining the on-chain Bitstring Status List Merkle root that enables the CredentialFreshness ZK circuit to prove credential validity and non-revocation in constant time without revealing credential content.
The service operates as a trusted component in v1 deployments, with a defined migration path to decentralised threshold operation in v2. Enterprise deployments may self-host the anchoring service using the open-source reference implementation and Docker images provided in the Attestara SDK.

Section X.2 — Architecture
The anchoring service consists of four internal components:
Status List Monitor. Polls the registered Bitstring Status List credential endpoint at a configurable interval, verifies the DataIntegrityProof signature on each fetch, and detects changes in the bitstring state by comparing against the previously cached version.
Merkle Tree Processor. On detecting a change or on heartbeat trigger, recomputes the Poseidon Merkle tree over the current bitstring, generates the SHA-256 bridging computation for on-chain compatibility, and produces the new Merkle root.
Transaction Manager. Submits the computed Merkle root to the CredentialRegistry contract on Arbitrum, manages gas pricing, monitors transaction confirmation, and handles resubmission on failure.
Health Monitor. Maintains operational metrics including last successful anchor timestamp, consecutive failure count, current gas price, and transaction confirmation latency. Exposes a health endpoint for enterprise monitoring integration.

Section X.3 — Anchoring Triggers
The anchoring service initiates a root update under two conditions:
Trigger 1 — Revocation Event. When the Status List Monitor detects a change in the Bitstring Status List credential — specifically any bit transitioning from 0 to 1 indicating revocation — an immediate anchor is initiated regardless of the heartbeat interval. The anchor transaction must be submitted within 5 seconds of detection. If the transaction fails, resubmission occurs at 10-second intervals with a maximum of three attempts before escalating to the failure handling procedure defined in Section X.6.
Trigger 2 — Heartbeat Interval. An anchor is initiated at the configured heartbeat interval regardless of whether any status change has occurred. The default heartbeat interval is 60 seconds. Heartbeat anchors maintain root freshness for ongoing negotiation sessions even when no revocations have occurred.
If a revocation event trigger and a heartbeat trigger coincide within a 5-second window, only one anchor transaction is submitted covering both triggers.

Section X.4 — Freshness Tiers
The anchoring service supports three freshness tiers. The tier is specified in the Authority Credential at issuance time by the Principal and cannot be modified by the Agent or the counterparty.
High Security Tier. Maximum staleness: 30 seconds. Heartbeat interval: 20 seconds. Applicable to financial services commitments, high-value procurement, and any deployment where a 90-second revocation window creates unacceptable risk. Estimated daily anchoring cost at current Arbitrum gas prices: approximately $173 per day.
Standard Tier. Maximum staleness: 90 seconds. Heartbeat interval: 60 seconds. Default for enterprise procurement and general B2B negotiation. Estimated daily anchoring cost: approximately $57.60 per day.
Low Frequency Tier. Maximum staleness: 300 seconds. Heartbeat interval: 240 seconds. Applicable to low-risk negotiation environments where anchoring cost is a constraint. Not recommended for financial services or high-value commitments. Estimated daily anchoring cost: approximately $14.40 per day.
The on-chain verifier enforces the freshness requirement by rejecting any CredentialFreshness proof where the status_list_root public input was anchored more than the tier-specified maximum staleness before the proof's currentTimestamp public input.

Section X.5 — Merkle Tree Construction
The anchoring service constructs the Merkle tree as follows:
The Bitstring Status List credential is fetched from its registered HTTPS endpoint and its DataIntegrityProof signature is verified before any processing. If signature verification fails, the fetch is treated as a failure and the previous cached state is retained.
The bitstring is decompressed from its GZIP-encoded form as specified in W3C Bitstring Status List v1.0. Each bit in the decompressed bitstring becomes a leaf in the Poseidon Merkle tree, with 0 representing an active credential and 1 representing a revoked credential.
The Poseidon Merkle root is computed over the full decompressed bitstring. For a standard 2^16 entry status list this produces approximately 12,000 constraints in the CredentialFreshness circuit as documented in the whitepaper.
The SHA-256 bridging computation recomputes the SHA-256 root from the Poseidon Merkle leaves. This value is what is submitted to the CredentialRegistry contract on-chain. The circuit verifies that the recomputed SHA-256 root matches the status_list_root anchored on-chain, as described in Section 4.4 of the whitepaper.

Section X.6 — Failure Handling
Failure handling is defined separately for each failure mode.
Status List Endpoint Unavailable. If the Bitstring Status List credential endpoint is unreachable for more than the tier-specific maximum staleness period, the anchoring service enters a degraded state. In degraded state:

High security tier: All CredentialFreshness proof verifications fail immediately. Active sessions are suspended. Principals are notified via the portal dashboard and configured alert channels.
Standard tier: Active sessions are suspended after the maximum staleness period. New sessions cannot be initiated. Principals are notified.
Low frequency tier: Active sessions are suspended after 600 seconds of unavailability. New sessions cannot be initiated. Principals are notified.

Sessions suspended due to anchoring unavailability may resume when service is restored provided the session anchor on-chain has not expired and the suspended turn's proof bundle is regenerated with the fresh status_list_root.
Transaction Submission Failure. If the anchor transaction fails to submit due to network issues or gas pricing, the Transaction Manager retries at 10-second intervals with gas price incrementing by 10% per retry. After three failed attempts, an alert is raised to the configured monitoring endpoint. After five failed attempts, the service enters the degraded state described above.
Transaction Confirmation Timeout. If a submitted transaction is not confirmed within 120 seconds, the Transaction Manager checks whether the transaction was included in a block with a different hash — indicating a reorg — and resubmits if necessary. If the transaction has been pending for more than 300 seconds, it is cancelled and resubmitted with current gas pricing.
DataIntegrityProof Verification Failure. If the fetched Bitstring Status List credential fails signature verification, the fetch is discarded and the previous cached state is retained. Three consecutive verification failures trigger an alert and the anchoring service enters the degraded state. This failure mode may indicate a compromised status list endpoint and should be treated as a security event requiring manual investigation.

Section X.7 — Self-Hosting Requirements
Enterprise deployments self-hosting the anchoring service must satisfy the following minimum requirements:
Infrastructure. A dedicated compute instance with guaranteed uptime SLA of 99.9% or higher. The anchoring service must not share compute resources with other workloads that could cause resource contention affecting anchor timing.
Key Management. The Ethereum private key used for anchor transaction signing must be stored in an HSM or equivalent key management system. The key must have sufficient ETH balance on Arbitrum to sustain anchoring operations for a minimum of 30 days without manual top-up — approximately 0.5 ETH at current gas prices for standard tier deployments.
Monitoring. The health endpoint must be integrated with the enterprise's existing monitoring infrastructure. Alerting must be configured for the degraded state, consecutive failure escalation, and DataIntegrityProof verification failures.
Network. Outbound access to the configured Bitstring Status List endpoint, the Arbitrum RPC endpoint, and the Alchemy fallback RPC endpoint. Inbound access to the health endpoint from the enterprise monitoring system.
Audit Logging. All anchor transactions, failure events, and status list fetches must be logged with timestamps to an append-only audit log retained for a minimum of 5 years to satisfy EU AI Act Article 12 and DORA record-keeping obligations.

Section X.8 — Operational Metrics
The anchoring service exposes the following metrics for monitoring and compliance reporting:

Last successful anchor timestamp and transaction hash
Current on-chain root value and age in seconds
Consecutive fetch failures and consecutive transaction failures
Total anchors submitted in the current 24-hour period
Current ETH balance of the signing key
Average transaction confirmation latency over the last 100 anchors
Degraded state entry and exit events with timestamps and causes

These metrics are available via the health endpoint and are surfaced in the Attestara portal dashboard under the Compliance Reports section.

Dimension 2 — Technical Architecture
The technical design of the threshold system needs to be specified completely before any implementation starts. The key components:
Consensus Mechanism
The five anchoring nodes need to agree on the Merkle root before any transaction is submitted on-chain. There are two approaches:
Option A — Off-chain consensus, single on-chain submission. The five nodes communicate off-chain, agree on the root through a lightweight consensus protocol, and one designated node submits the transaction. The other four co-sign the root off-chain as attestation. The CredentialRegistry contract verifies the co-signatures before accepting the root.
Option B — Independent submission, on-chain threshold. Each node independently computes the Merkle root and submits it to the CredentialRegistry contract. The contract accepts the root once three matching submissions have been received.
Recommendation: Option A for v1 production. Option B requires the CredentialRegistry contract to handle multiple submissions and match them, which adds contract complexity and gas cost. Option A is cheaper on-chain and the off-chain consensus adds negligible latency for a 5-node network. The co-signature verification on-chain is a small additional gas cost that's worth the security guarantee.
Consensus Window
The time within which all participating nodes must respond during consensus. Given the heartbeat intervals:

High Security tier: 5-second consensus window
Standard tier: 15-second consensus window
Low Frequency tier: 60-second consensus window

If 3-of-5 nodes don't reach consensus within the window, the previous root remains valid and an alert is raised. This is preferable to submitting a root that only 2 nodes have agreed on.
Node Communication Protocol
The five nodes need a secure, authenticated communication channel for off-chain consensus. Options:

A dedicated libp2p network between node operators
A simple HTTPS-based coordinator service
DIDComm v2 — which would be architecturally elegant given the rest of the protocol

DIDComm v2 for node consensus communication is worth serious consideration. Each anchoring node has a DID. Consensus messages are authenticated DIDComm messages. This reuses existing protocol infrastructure and makes the consensus communication auditable through the same mechanisms as agent negotiations.

5. Technical Architecture
5.1 Consensus Mechanism
The TAN uses off-chain consensus with single on-chain submission. This approach minimises on-chain gas costs while maintaining cryptographic security guarantees.
Consensus flow:
1. Trigger fires (revocation event or heartbeat) at any node
2. Triggering node computes Merkle root and broadcasts to all other nodes
3. Each receiving node independently verifies the root computation
4. Nodes that agree respond with a co-signature within the consensus window
5. Once 3-of-5 co-signatures are collected, the triggering node submits 
   the root on-chain with all co-signatures
6. CredentialRegistry contract verifies co-signatures and updates root
7. All nodes record the confirmed anchor in their audit logs
Consensus window by tier:
TierMaximum StalenessHeartbeat IntervalConsensus WindowHigh Security30 seconds20 seconds5 secondsStandard90 seconds60 seconds15 secondsLow Frequency300 seconds200 seconds60 seconds
Deduplication: If a revocation event trigger and heartbeat trigger fire within a 5-second window, one consensus round covers both.
5.2 Node Communication Protocol
Inter-node consensus communication uses DIDComm v2 authenticated messaging. Each anchoring node has a DID registered on Arbitrum via ERC-1056. Consensus messages are authenticated DIDComm messages, making all consensus communication auditable through the same mechanisms as agent negotiations.
This is architecturally consistent with the broader Attestara protocol and avoids introducing a separate authentication mechanism for node communication.

5.3 CredentialRegistry Contract Updates
The existing CredentialRegistry contract requires the following updates for threshold production:

pragma solidity ^0.8.19;

contract CredentialRegistry {
    
    // Threshold configuration
    address[] public anchoringNodes;
    uint256 public threshold; // 3
    uint256 public constant MAX_NODES = 10;
    
    // Current anchored root per tier
    mapping(uint8 => bytes32) public currentRoot; // tier => root
    mapping(uint8 => uint256) public lastAnchorTimestamp; // tier => timestamp
    
    // Governance
    mapping(address => bool) public isAnchoringNode;
    mapping(bytes32 => mapping(address => bool)) public nodeVotes;
    mapping(bytes32 => uint256) public voteCount;
    
    // Events
    event RootAnchored(
        bytes32 indexed merkleRoot, 
        uint8 tier,
        uint256 timestamp,
        address submitter
    );
    event NodeAdded(address indexed node);
    event NodeRemoved(address indexed node);
    
    // Submit anchor root with co-signatures
    function submitAnchorRoot(
        bytes32 merkleRoot,
        uint8 tier,
        uint256 anchorTimestamp,
        bytes[] calldata coSignatures,
        address[] calldata coSigners
    ) external {
        require(isAnchoringNode[msg.sender], "Not authorised node");
        require(coSignatures.length >= threshold - 1, "Insufficient signatures");
        require(
            block.timestamp - anchorTimestamp <= _freshnessWindow(tier),
            "Anchor timestamp too old"
        );
        
        // Verify each co-signature
        bytes32 messageHash = keccak256(
            abi.encodePacked(merkleRoot, tier, anchorTimestamp)
        );
        for (uint i = 0; i < coSigners.length; i++) {
            require(isAnchoringNode[coSigners[i]], "Co-signer not authorised");
            require(
                _verifySignature(messageHash, coSignatures[i], coSigners[i]),
                "Invalid co-signature"
            );
        }
        
        // Update root
        currentRoot[tier] = merkleRoot;
        lastAnchorTimestamp[tier] = anchorTimestamp;
        
        emit RootAnchored(merkleRoot, tier, anchorTimestamp, msg.sender);
    }
    
    // Governance — add node requires supermajority
    function proposeNodeAddition(address newNode) external {
        require(isAnchoringNode[msg.sender], "Not authorised");
        require(anchoringNodes.length < MAX_NODES, "Maximum nodes reached");
        // Voting mechanism — requires 4-of-5 approval
    }
    
    // Freshness window per tier
    function _freshnessWindow(uint8 tier) internal pure returns (uint256) {
        if (tier == 0) return 30;   // High Security
        if (tier == 1) return 90;   // Standard
        if (tier == 2) return 300;  // Low Frequency
        revert("Invalid tier");
    }
}
5.4 Self-Healing and Node Failure
Failure detection: Each node monitors the consensus participation of all other nodes. A node is flagged as potentially unavailable after missing 10 consecutive consensus rounds.
Network continuity: The network continues operating with 3+ active nodes. Two simultaneous node failures maintain operation. Three simultaneous failures suspend the network.
Recovery procedure:

Flagged node: Automatic alert to all operators and Littledata
Unavailable less than 4 hours: Node rejoins automatically on recovery
Unavailable 4-24 hours: Manual reconnection procedure required
Unavailable more than 24 hours: Governance review initiated — node may be temporarily suspended from the active set

Suspension procedure: A temporarily suspended node is removed from the active signing set via governance vote. The threshold adjusts to 2-of-4 during suspension with a maximum suspension period of 30 days. If the node is not recovered within 30 days, replacement recruitment begins.
5.5 Key Management Requirements
Key generation ceremony: Each node operator generates their anchoring key in a documented ceremony. The ceremony must:

Be performed on air-gapped hardware
Generate the key directly into the HSM — never in plaintext on any connected system
Produce a ceremony record documenting the hardware, procedure, and witnesses
Result in the public key being published to the Attestara transparency log

HSM requirements: Any HSM meeting FIPS 140-2 Level 3 or equivalent certification is acceptable. Supported HSM types include AWS CloudHSM, Azure Dedicated HSM, Thales Luna, and YubiHSM 2.
Key rotation: Node operators must rotate anchoring keys at least annually. Key rotation is performed through the ERC-1056 delegate rotation mechanism without changing the node's root DID. All other operators must be notified minimum 48 hours before rotation.

6. Staged Migration Plan
6.1 Migration Stages
Migration from centralised to threshold production follows four defined stages with explicit entry and exit criteria.
Stage 1 — Shadow Mode (30 days)
The threshold network runs in parallel with the centralised service. Threshold nodes compute Merkle roots and reach consensus but do not submit on-chain. Their computed roots are compared against the centralised service submissions.
Entry criteria: All five nodes operational and passing health checks
Exit criteria: 30 days of shadow operation with less than 0.1% root discrepancy rate
Stage 2 — Dual Submission (30 days)
Both the centralised service and the threshold network submit roots on-chain. The CredentialRegistry accepts either. Discrepancies are monitored and investigated.
Entry criteria: Stage 1 exit criteria met
Exit criteria: 30 days of dual operation with zero unresolved discrepancies and threshold network transaction success rate above 99.5%
Stage 3 — Threshold Primary (30 days)
The threshold network is the primary anchoring mechanism. The centralised service runs as a fallback — it only submits if the threshold network fails to anchor within 1.5x the tier freshness window.
Entry criteria: Stage 2 exit criteria met
Exit criteria: 30 days of threshold-primary operation with centralised fallback not triggered more than twice
Stage 4 — Full Threshold
The centralised service is decommissioned. The Littledata centralised node is optionally retained as one of the five threshold nodes or replaced by an independent operator depending on the governance decision at that time.
Entry criteria: Stage 3 exit criteria met and all active enterprise customers notified and migrated
Exit criteria: Centralised service decommissioned and confirmed inactive for 30 days
6.2 Rollback Procedure
At any stage, if exit criteria are not met or a critical incident occurs, rollback to the previous stage is executed as follows:

Stage 2 → Stage 1: Disable threshold on-chain submission, continue shadow mode, investigate
Stage 3 → Stage 2: Re-enable centralised submission as co-primary, investigate threshold failure
Stage 4 → Stage 3: Re-enable centralised service as fallback, root cause analysis required

Rollback decisions are made jointly by Littledata and a majority of active node operators.

Understanding The Problem Space First
Before writing the specification, we need to be precise about what makes cross-org session establishment fundamentally different from intra-org sessions.
In an intra-org session both agents share the same Principal's infrastructure. DID resolution, credential anchoring, and DIDComm routing all happen within a trusted boundary. The session establishment is essentially an internal API call with known endpoints.
In a cross-org session none of that applies. Agent A and Agent B:

Have never interacted before
Cannot trust each other's infrastructure
Cannot trust each other's DID resolution results without independent verification
Are representing genuinely competing commercial interests
May be subject to different regulatory jurisdictions
May be running on completely different agent frameworks

The session handshake must establish cryptographic trust between two parties who start with zero shared context and zero institutional trust in each other. This is the hardest trust establishment problem in the entire protocol and it needs to be specified at the same level of rigour as the ZK circuits.

The Five Problems The Handshake Must Solve
Before writing a single line of specification, define exactly what problems the handshake solves. Each problem maps to a specific mechanism in the specification.
Problem 1 — Agent Discovery.
How does Agent A find Agent B's DID and DIDComm service endpoint when they have never interacted? This is a pre-protocol problem but the specification needs to define the discovery mechanism or explicitly scope it out.
Problem 2 — DID Authentication.
How does each agent prove it controls its DID before the session begins? Possession of the DID is not sufficient — an impersonator who knows Agent B's DID but does not control its private key must be rejected.
Problem 3 — Credential Verification.
How does each agent verify that the counterparty holds a valid, unrevoked, on-chain anchored Authority Credential before committing to a session? This must happen without the credential being transmitted — the credential's mandate parameters are private.
Problem 4 — Session Binding.
How is the session cryptographically bound to both agents' DIDs such that neither can later deny participating? The session anchor must be established on-chain before any negotiation turns occur.
Problem 5 — Replay and Injection Prevention.
How does the handshake prevent an attacker from replaying a previous session invitation, injecting a fake session anchor, or substituting their own agent into an established session?

Architectural Decisions To Make First
Three decisions need to be locked before the specification is written.

Decision A — Agent Discovery Mechanism
The whitepaper currently assumes agents know each other's DIDs before initiating a session. For the testnet demo this is fine — you configure both agents manually. For production cross-org deployment this assumption breaks down.
Three options:
Option A — Out of band discovery.
Agent discovery happens through existing business relationship channels — email, API documentation, partner portal. The Attestara protocol begins after DIDs are exchanged. Discovery is explicitly out of scope.
Option B — DIF Universal Resolver plus DIDComm service endpoint advertising.
Organisations publish their agent DIDs and DIDComm service endpoints in a standardised way — either through a DIF-compatible agent registry or through well-known URIs on their domain. Agent A discovers Agent B by resolving their organisation's DID document.
Option C — Attestara Agent Registry.
A dedicated on-chain or off-chain registry where organisations register their agent DIDs by capability type, industry, and jurisdiction. Agent A queries the registry to find eligible counterparties.
Recommendation: Option A for v1 with Option B as a documented extension.
Option A is the pragmatically correct choice for v1. The bilateral pilot use cases you're targeting all involve known counterparties — a manufacturer and their supplier, two financial institutions negotiating a credit facility. Discovery through existing business channels is appropriate and keeps the protocol scope clean.
Option B should be documented as a planned extension because it's the mechanism that enables the broader agentic commerce use cases — where Agent A doesn't know in advance which suppliers to negotiate with. This is a Phase 3+ capability but should be referenced in the specification so enterprise architects can see the roadmap.
Option C is premature and creates a centralisation risk you've already avoided elsewhere in the protocol.

Decision B — Mutual Authentication Sequence
Two options for how mutual DID authentication is structured:
Option A — Sequential authentication.
Agent A authenticates to Agent B first, then Agent B authenticates to Agent A. Simpler but creates a brief window where one agent has authenticated without the other.
Option B — Simultaneous challenge-response.
Both agents issue challenges simultaneously and respond to each other's challenges in a single round trip. More complex but eliminates the asymmetric authentication window.
Recommendation: Option B — simultaneous challenge-response.
In an adversarial cross-org context the asymmetric window in Option A is a real attack surface. If Agent A authenticates to Agent B but Agent B's authentication fails, Agent A has revealed its identity and session intent to a potentially malicious counterparty without any reciprocal commitment. Simultaneous challenge-response eliminates this.

Decision C — On-Chain Anchor Timing
When exactly is the session anchored on-chain relative to the handshake?
Option A — Anchor before handshake.
Agent A creates the session anchor on-chain before sending the invitation. Agent B verifies the anchor exists before responding.
Option B — Anchor after mutual authentication.
Mutual authentication completes first, then Agent A creates the session anchor. Agent B verifies the anchor before the first negotiation turn.
Option C — Anchor jointly.
Both agents co-sign the session anchor transaction. The anchor only goes on-chain when both have committed.
Recommendation: Option B for v1, with Option C documented as a Phase 2 enhancement.
Option A creates a problem — Agent A incurs on-chain cost before knowing whether Agent B will respond or is even a valid counterparty. For standard tier deployments at $0.015 per session anchor this is acceptable but it creates a griefing attack surface where a malicious party could trigger many session creations against Agent A without responding.
Option B requires Agent B to trust that Agent A will create the anchor after authentication, which is reasonable given the mutual authentication has already committed both parties cryptographically.
Option C is the most secure but requires a joint transaction signing mechanism that adds complexity better deferred to Phase 2.

The Cross-Org Session Handshake Specification
With those decisions locked, here is the full specification.

Section 7.3a — Cross-Organisational Session Establishment
7.3a.1 Overview
Cross-organisational session establishment is the protocol sequence through which two agents representing different organisations establish a cryptographically bound, mutually authenticated negotiation session without requiring inter-party trust or shared infrastructure.
The cross-org handshake solves five distinct problems in sequence:

Agent discovery — establishing the counterparty's DID and DIDComm endpoint
Mutual DID authentication — proving control of DIDs through simultaneous challenge-response
Credential existence verification — confirming each agent holds a valid on-chain anchored Authority Credential without revealing mandate parameters
Session anchor creation — establishing an immutable on-chain session record before any negotiation turns occur
Replay and injection prevention — cryptographic mechanisms preventing session hijacking, replay, and man-in-the-middle attacks

7.3a.2 Scope and Assumptions
This section specifies the cross-org session establishment protocol for v1 conforming implementations.
In scope:

Mutual DID authentication
Credential existence verification
Session anchor creation and verification
Invite token generation and validation
Session establishment failure handling
Security properties of the completed handshake

Out of scope for v1:

Agent discovery — how Agent A obtains Agent B's DID and DIDComm service endpoint is a pre-protocol concern. v1 assumes DIDs are exchanged through existing business relationship channels. A standardised agent discovery extension based on DIDComm service endpoint advertising is planned for v2.
Joint session anchor co-signing — v1 uses single-party anchor creation by the session initiator. Dual-party anchor co-signing is a planned Phase 2 enhancement.
Multi-party sessions involving more than two agents.

Assumptions:

Both agents have valid, on-chain anchored Authority Credentials before session initiation
Both agents have resolvable DIDs via the DIF Universal Resolver or direct Ethereum RPC
Both agents have access to a DIDComm v2 service endpoint — either the Attestara Session Relay or a self-hosted relay
The initiating agent (Agent A) has obtained the counterparty agent's DID through pre-protocol channels

7.3a.3 Protocol Participants
Agent A (Initiator). The agent that initiates the session. Responsible for generating the session ID, creating the on-chain session anchor, and sending the initial session invitation.
Agent B (Responder). The agent that receives and responds to the session invitation. Responsible for verifying the session anchor and completing the mutual authentication before the first negotiation turn.
Principal A. The organisation that issued Agent A's Authority Credential. Notified of session initiation and termination events through the portal dashboard.
Principal B. The organisation that issued Agent B's Authority Credential. Notified of session initiation and termination events through the portal dashboard.
Session Relay. The encrypted transport infrastructure carrying DIDComm v2 messages between agents. Operates as a pass-through — it sees session metadata but never message content.
CredentialRegistry Contract. The on-chain contract that holds credential hash anchors and session anchors. Both agents query this contract independently during the handshake.
7.3a.4 Pre-Handshake Requirements
Before the handshake begins, Agent A must independently verify:
Requirement 1 — Counterparty DID resolvability.
Agent A resolves Agent B's DID through the Ethereum RPC endpoint via the ERC-1056 registry. The resolved DID document must contain a valid Ed25519 verification method and a DIDComm v2 service endpoint. If either is absent the session cannot proceed and Agent A must surface an error to Principal A.
Requirement 2 — Counterparty credential anchor existence.
Agent A queries the CredentialRegistry contract to confirm that a credential hash anchor exists for Agent B's DID. This confirms Agent B holds at least one issued Authority Credential. It does not reveal the credential content, mandate parameters, or revocation status — only that an anchor exists.

Note: Agent A verifying credential anchor existence before sending an invitation prevents invitation spam to agents that have no valid credentials. It does not guarantee the credential is currently valid — full credential freshness verification happens during the mutual authentication phase.

Requirement 3 — Session Relay reachability.
Agent A confirms the Session Relay endpoint is reachable and returns a valid health response. If the relay is unreachable, Agent A should attempt direct P2P DIDComm delivery to Agent B's service endpoint before aborting.
7.3a.5 Handshake Message Flow
The cross-org session handshake consists of five message exchanges across three phases.

   |--- [1] SESSION-INVITE ---->|----------forward------------->|

sessionParams — Non-cryptographic session parameters including maximum negotiation turns, session expiry timestamp, and the negotiation schema identifier defining what parameters will be negotiated.
Pre-send requirements for Agent A:
Before sending Message 1, Agent A must:

Generate sessionId, challengeA, and inviteToken using a cryptographically secure random number generator
Submit the session anchor transaction to the CommitmentContract and wait for confirmation — minimum 1 block confirmation on Arbitrum before sending the invitation
Sign inviteToken with Agent A's DID private key — the signature is included in the DIDComm v2 envelope, not in the message body
Encrypt the message body using Agent B's public key resolved from their DID document via ECDH-1PU + XC20P

Session Relay processing:
The Session Relay forwards the encrypted message to Agent B's DIDComm service endpoint. The relay records the session metadata — Agent A DID, Agent B DID, session ID, timestamp — but cannot decrypt the message body.
7.3a.7 Phase 2 — Simultaneous Challenge-Response
Message 2: CHALLENGE-RESPONSE
Upon receiving Message 1, Agent B independently verifies the session invitation before responding. Agent B then sends a combined challenge-response that simultaneously proves its DID control, provides its own challenge to Agent A, and confirms its credential status.
Agent B verification steps before sending Message 2:
Step 1 — Verify invite token signature.
Agent B resolves Agent A's DID document and verifies the invite token signature in the DIDComm v2 envelope was produced by Agent A's DID key.
Step 2 — Verify session anchor on-chain.
Agent B independently queries the CommitmentContract on Arbitrum to confirm the session anchor transaction referenced in sessionAnchorTx exists, is confirmed in the block referenced in sessionAnchorBlock, and contains Agent A's DID as the initiating agent and the sessionId as the session identifier.

Critical: Agent B must perform this verification independently using its own Ethereum RPC endpoint — not by trusting any data provided by Agent A. The purpose of the on-chain anchor is precisely to provide a neutral verification point that neither party controls.

Step 3 — Verify Agent A credential anchor.
Agent B queries the CredentialRegistry contract to confirm the agentACredentialAnchor provided by Agent A exists on-chain and is associated with Agent A's DID. This confirms Agent A holds at least one issued credential. It does not verify the credential's current revocation status — full CredentialFreshness verification happens during negotiation turns.
Step 4 — Verify invite token is unused.
Agent B checks its local invite token registry to confirm this inviteToken has not been previously used. If the token has been used, Agent B rejects the invitation with error code attestara.session.token-replayed.
Step 5 — Verify session expiry.
Agent B confirms created_time is within the last 300 seconds and expires_time has not passed. If either check fails, Agent B rejects with attestara.session.invite-expired.
Step 6 — Confirm freshness tier capability.
Agent B confirms its anchoring service is currently operating within the freshness tier specified in freshnessRequirement. If the anchoring service is in a degraded state that prevents meeting the required freshness, Agent B rejects with attestara.session.freshness-unavailable.
If all six verification steps pass, Agent B sends Message 2:

7.3a.9 Session Establishment Security Properties
Upon successful completion of the five-message handshake, the following security properties are guaranteed:
Mutual authentication. Both agents have proved control of their respective DIDs through the simultaneous challenge-response mechanism. Neither agent accepted the other's identity on the basis of claimed DID alone.
Credential existence. Both agents have independently verified that the counterparty holds at least one on-chain anchored Authority Credential. Neither agent accepted this on the counterparty's word — both queried the CredentialRegistry independently.
Session binding. The session is cryptographically bound to both agents' DIDs through the on-chain session anchor. Neither agent can later deny participation in the session without producing a valid cryptographic proof that they did not control the DID associated with the session.
Replay prevention. The single-use invite token, the message timing checks, and the handshake hash collectively prevent replay attacks. A replayed Message 1 will be rejected by the invite token check. A replayed Message 2 will be rejected by the timing check. A substituted session context will be detected by the handshake hash verification.
Man-in-the-middle prevention. The mutual DID authentication using challenge-response prevents a relay operator or network attacker from impersonating either agent. The Session Relay can see session metadata but cannot impersonate either agent without controlling their DID private keys.
Forward secrecy. DIDComm v2 ECDH ephemeral key exchange provides forward secrecy for all negotiation turn messages. Compromise of a DID long-term key after session completion does not expose the content of past negotiation turns.
What the handshake does not guarantee:
The cross-org handshake establishes mutual authentication and session binding. It does not:

Verify that the counterparty's credential is currently unrevoked — this is verified per-turn through the CredentialFreshness ZK circuit
Verify that the counterparty's agent is operating within its mandate — this is enforced per-turn through the MandateBound and ParameterRange ZK circuits
Guarantee the counterparty's agent is reasoning honestly — as noted in Section 8.4, Attestara governs proposals within mandate bounds, not the AI model's reasoning process

All handshake messages must be processed and responses sent within the following windows:

The 30-second window for Message 2 accounts for the time Agent B requires to independently verify the session anchor on-chain — Arbitrum block times average 250ms but the RPC query and verification logic adds latency that varies by enterprise deployment.
The 10-second windows for Messages 3-5 assume verification is faster once the on-chain anchor is confirmed — subsequent verifications are simpler signature and hash checks that should complete in under 1 second in normal operation.
7.3a.12 Session Key Establishment
The cross-org handshake establishes a shared session context but uses DIDComm v2's native ECDH-1PU key agreement for message encryption rather than a separate session key exchange. This is intentional — DIDComm v2 provides forward secrecy through ephemeral key generation per message without requiring a separate key exchange protocol.
The sessionKeyCommitment in Message 3 is a commitment to the ephemeral key Agent A will use for the first negotiation turn message. This allows Agent B to verify that the first negotiation turn message is encrypted by the same entity that completed the handshake, preventing a session hijacking attack where an attacker who observed the handshake attempts to inject a fake first turn.

7.3a.14 v2 Extensions — Documented for Implementers
The following extensions to the cross-org session establishment protocol are planned for v2 and are documented here so that v1 implementations are designed to accommodate them without breaking changes:
Agent discovery via DIDComm service endpoint advertising. Organisations will be able to publish their agent DIDs and capability schemas at well-known URIs on their domains, enabling Agent A to discover eligible counterparties without prior business contact.
Joint session anchor co-signing. Both agents will co-sign the session anchor transaction in a two-phase commit, eliminating the v1 asymmetry where Agent A bears the anchor creation cost before Agent B has committed to the session.
Delegation chain verification. For enterprise deployments where an agent acts on behalf of multiple principals through a delegation chain, the handshake will include verification of the full delegation chain rather than just the immediate Authority Credential.
Session resumption. Suspended sessions — due to anchoring service unavailability as specified in the anchoring service section — will be resumable through a lightweight re-authentication using the stored handshakeHash as a session context proof, without requiring a full five-message handshake.
